迁移策略：5 个 Phase，逐层剥离

原则：从叶子节点开始（零内部依赖的包先拆），每步可独立编译测试，失败可回滚。

Phase 1 — 零依赖叶子包（最安全，最先动）

provider/
移入：openai.go + claude provider
Deps：仅外部 SDK
Importers：~15+ 文件
风险：低
observation/
移入：assertion.go
Deps：零（纯逻辑）
Importers：executor, observer
风险：极低
recording/
移入：recorder.go + replayer.go
Deps：仅 store/
Importers：cognitive.go, gateway
风险：低
healing/
移入：auto_heal.go
Deps：仅 tool/ (ToolCall)
Importers：executor, cognitive
风险：低
checkpoint/
移入：checkpoint.go
Deps：仅 store/
Importers：cognitive.go, gateway
风险：极低

Phase 2 — 轻量依赖包（只依赖外部 + Phase1 叶子）

planning/
移入：plan.go + planner_tree.go + TaskPlan/SubTask 类型
Deps：provider/, config/
风险：中
perception/
移入：perceive.go + project_scanner + git_context + context_budget + failure_context
Deps：memory/, knowledge/
风险：中
reflection/
移入：reflect.go
Deps：provider/, memory/
风险：低

Phase 3 — 核心包（依赖较多，最需小心）

execution/
移入：act.go + concurrent.go (Executor + ToolExecutor)
Deps：tool/, store/, session/, hook/, channel/, healing/, observation/
风险：高 — 被 gateway 大量引用
runtime/
移入：runtime.go + stream.go + compression.go + context_manager.go + speculative.go
Deps：provider/, tool/, session/, store/
风险：高 — 简单模式入口

Phase 4+5 — 子代理 + 瘦身

Phase 4: subagent/
移入：subagent.go + agent_manager.go + agent_tool.go + team相关
Deps：runtime/, execution/, tool/, session/
Phase 5: 收缩 agent/
只剩 cognitive.go (~250行) + cognitive_types.go + cognitive_prompts.go
其余全部迁出。tool_cache.go → execution/
sidechain.go → runtime/
每 Phase 检查清单： ① go build ./...   ② go test ./...   ③ golangci-lint   ④ 确认 gateway/ cmd/ eval/ 无断引用
