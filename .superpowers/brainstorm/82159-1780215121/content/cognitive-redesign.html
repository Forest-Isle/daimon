Cognitive Agent 重设计 · 当前问题 vs 新架构

当前 cognitive.go 1562 行。问题不在 5 阶段概念——而在实现方式。

🔴 当前设计 · 4 个结构性问题

1. Phase 是具体 struct，不是接口
Perceiver/Planner/Executor/Observer/Reflector 都是具体类型。
不能替换、不能 mock、不能组合。

2. 30 个可选依赖通过 nil-check 散落各处
if ca.selfHealEngine != nil { ... }
if ca.replayRecorder != nil { ... }
if ca.planMode != nil { ... }
加一个 feature → 加一个字段 + N 个 nil 检查点

3. Run() 方法是一个状态机噩梦
正常循环 + replan + checkpoint 恢复 + plan-mode 审批等待
+ 413 响应压缩 + 自主循环模式 — 全塞在一个方法里

4. 控制流和业务逻辑耦合
"replan 几次后放弃"、"checkpoint 什么时候存"、
"plan-mode 审批怎么等待" — 这些是横切关注点，
不应该嵌在 phase 执行逻辑里。

🟢 新设计 · Phase 接口 + Pipeline Middleware

1. Phase 变成接口
type Phase interface {
Name() string
Run(ctx, state) (*Output, error)
}
5 个实现：PerceivePhase, PlanPhase, ActPhase,
ObservePhase, ReflectPhase

2. 横切关注点变成 Pipeline Middleware
• ReplanMiddleware — 失败→回到 PLAN
• CheckpointMiddleware — 每阶段后自动保存
• PlanModeMiddleware — PLAN 后等待审批
• SelfHealMiddleware — ACT 工具错误修复
• ReplayMiddleware — 记录所有阶段转换
• CompressionMiddleware — 413 响应自动压缩

3. 状态转换表替代硬编码循环
PERCEIVE→PLAN→ACT→OBSERVE→{REFLECT|PLAN}→DONE
每条转换由 PhaseOutput.NextPhase 决定

4. 内聚到 core/cognitive/ 包
直接依赖 core.Agent 的 Provider + Tool 接口
Gateway 从 55 字段自然瘦身

迁移策略 · Strangler Fig（绞杀榕模式）

Step 1: 在 internal/core/cognitive/ 定义 Phase 接口 + Pipeline runner
Step 2: 写 adapter 包装现有 Perceiver/Planner/Executor/Observer/Reflector 为 Phase 实现
Step 3: gateway.go 同时持有旧 CognitiveAgent 和新 Pipeline，feature flag 切换
Step 4: 逐个将 adapter 替换为真正的 Phase 实现
Step 5: 删除 agent/cognitive.go 的编排逻辑，只保留 phase 内部实现
Step 6: 删除 adapter 层

每一步独立可验证、可回滚。不影响现有功能。

风险与范围

这是整个修复计划中最大的单一体力活。
预计新增: internal/core/cognitive/ (6-8 个新文件, ~2000 行)
预计修改: gateway.go（瘦身 30%）, agent/cognitive.go（删除编排逻辑）
预计删除: agent/ 中的编排代码（~400 行）

需要独立 worktree，不能和其他改动混在一起。
这一步放在轨道 1 的第一优先级——因为其他拆分都依赖 agent 结构稳定。
