# 11 - 基础设施（Config / Session / Store / Scheduler / UserDir）

## 一、Config 配置系统

### 文件

```
internal/config/
├── config.go    # 配置加载 + 类型定义
└── validate.go  # 配置验证
```

### 配置加载流程

```
config.Load(path)
    │
    ├── 1. os.ReadFile(path)
    ├── 2. ExpandEnv(data)       # ${VAR} → 环境变量值
    ├── 3. defaultConfig()       # 填充默认值
    ├── 4. yaml.Unmarshal()      # YAML 解析
    └── 5. validate()            # 配置验证
```

### 配置层次

```
1. 代码默认值 (defaultConfig())        ← 最低优先级
2. 项目配置 (configs/ironclaw.yaml)
3. 用户覆盖 (~/.IronClaw/config.yaml)  ← 最高优先级
```

### 配置结构概览

```go
type Config struct {
    LLM       LLMConfig       // Claude API 配置

```

### 环境变量展开

```yaml
# 配置文件中
llm:
  api_key: "${ANTHROPIC_API_KEY}"

# 运行时自动替换为环境变量值
```

正则：`\$\{([^}]+)\}`，未匹配的占位符保持原样。

## 二、Session 会话管理

### 文件

```
internal/session/
├── session.go   # Session 结构体
└── manager.go   # Manager (CRUD + 持久化)
```

### Session 结构

```go
type Session struct {
    ID        string
    Channel   string            // "telegram" | "tui"
    ChannelID string            // 唯一标识对话
    Messages  []Message
    CreatedAt time.Time
    UpdatedAt time.Time
    Metadata  map[string]string
    mu        sync.Mutex
}

type Message struct {
    ID        string
    Role      string  // user | assistant | system | tool_use | tool_result
    Content   string
    ToolName  string
    ToolInput string
    CreatedAt time.Time
}
```

### Manager 操作

```go
type Manager struct { ... }

func (m *Manager) Get(ctx, channel, channelID) (*Session, error)
    // 获取或创建会话

func (m *Manager) Persist(ctx, sess) error
    // 持久化到 SQLite

func (m *Manager) Reset(ctx, channel, channelID) error
    // 重置会话（/new 或 /start 命令）
```

### 工具执行日志

```go
func LogToolExecution(ctx, db, sessionID, toolName, input, output, status, duration)
```

记录每次工具调用的输入、输出、状态和耗时，用于审计和 RL 训练。

### SQLite 配置

```go
func Open(path string) (*DB, error)
    │
    ├── sql.Open("sqlite3", path)
    ├── PRAGMA journal_mode=WAL     # WAL 模式（并发读）
    ├── PRAGMA busy_timeout=5000    # 锁等待 5s
    ├── 嵌入式迁移 (//go:embed migrations/*.sql)
    │   └── 按文件名字母顺序执行
    │       CREATE TABLE IF NOT EXISTS → 幂等
    ├── FTS5 可用性探测
    │   └── 不可用时降级为 LIKE 查询
    └── 返回 *DB
```

### FTS5 降级策略

```
FTS5 可用?
    ├── YES → 使用 FTS5 虚拟表 (BM25 排序)
    └── NO  → 使用 LIKE '%query%' (无排序)
```

## 四、Scheduler 定时调度

### 文件

```
internal/scheduler/
├── scheduler.go  # 调度器
└── task.go       # Task 定义
```

### 结构

```go
type Task struct {
    Channel   string
    ChannelID string
    Prompt    string
    Schedule  string  // cron 表达式
}

type Scheduler struct {
    db           *store.DB
    pollInterval time.Duration
    handler      func(ctx, Task)  // 执行回调
}
```

### 工作原理

```
Start(ctx):
    │
    └── 后台 goroutine:
        每 pollInterval 轮询 DB
        │
        ├── 查找到期任务
        └── handler(task) → gateway.handleInbound()
            将任务作为消息注入正常处理管线
```

调度任务被路由为来自 `scheduler` 用户的普通消息，复用完整的 Agent 处理流程。

## 设计亮点

1. **配置层次化**：默认值 → 项目配置 → 用户覆盖，清晰的优先级
2. **嵌入式迁移**：SQL 文件通过 `//go:embed` 编译进二进制
3. **幂等迁移**：所有 `CREATE TABLE IF NOT EXISTS`
4. **WAL 模式**：允许并发读取
5. **Soul.md / Memory.md**：人类可读的人格和规则定义
6. **调度器复用**：定时任务通过 handleInbound 复用 Agent 管线
