# IronClaw 子代理系统架构文档

> 基于 claude-code 子代理架构优化，适配 Go 生态

---

## 一、系统概述

IronClaw 的子代理（Subagent）系统允许编排者代理（Orchestrator Agent）将专业化任务委派给子代理执行，形成多代理协作框架。经过三期优化，系统现已具备以下核心能力：

| 能力 | 说明 | 实现阶段 |
|------|------|----------|
| **三种执行模式** | Spawn（独立）、Fork（继承上下文）、Background（后台异步） | 第 1/2 期 |
| **并行/DAG 调度** | AgentOrchestrator 支持并行执行和依赖拓扑排序 | 第 1 期 |
| **上下文隔离** | SubagentContext 精确控制共享与隔离边界 | 第 1 期 |
| **Prompt Cache** | 系统提示词去重缓存，减少重复构建开销 | 第 2 期 |
| **Sidechain 记录** | 子代理执行历史独立记录，支持恢复和审计 | 第 2 期 |
| **多执行后端** | InProcess / Subprocess / Docker 三种后端抽象 | 第 3 期 |
| **权限分级** | bypass / accept_edits / bubble 三级权限模型 | 第 3 期 |
| **生命周期钩子** | OnStart / OnComplete / OnError / OnTimeout / OnToolCall | 第 3 期 |
| **Per-Agent MCP** | 每个子代理可配置独立的 MCP 服务器 | 第 3 期 |

## 三、核心组件详解

### 3.1 AgentSpec — 子代理规格定义

`AgentSpec` 是子代理的完整配置，支持 YAML 文件定义和内联定义两种方式。

**文件位置**：`internal/agent/spec.go`

**完整字段列表**：

```yaml
# ~/.IronClaw/agents/code-writer.yaml
name: code-writer                       # 唯一标识符（必填）
description: 编写和测试代码变更             # 描述（必填）
system_prompt: |                        # 自定义系统提示词
  你是一个代码实现专家...
model: claude-sonnet-4-20250514         # LLM 模型覆盖（可选）
max_tokens: 8000                        # 最大输出 token（可选）
max_iterations: 10                      # 迭代上限（默认 5）
timeout: 300s                           # 执行超时（默认 120s）
mode: simple                            # "simple" | "cognitive"
requires_approval: false                # 是否需要用户批准
max_retries: 0                          # 失败重试次数

# --- 执行模式 ---
execution_mode: fork                    # "spawn"（默认）| "fork" | "background"
permission_mode: accept_edits           # "" | "bubble" | "accept_edits" | "bypass"
inherit_context: true                   # fork 模式自动设为 true
max_output_tokens: 16000                # 输出 token 限制

# --- 执行后端 ---
backend: in_process                     # "in_process"（默认）| "subprocess" | "docker"

# --- 工具白名单 ---
tools:                                  # 空 = 所有工具（agent_* 始终排除）
  - bash
  - file_read
  - file_write

# --- 路由标签 ---
tags: [code, implementation]

# --- 生命周期钩子 ---
hooks:
  on_start:
    - type: log
      message: "代理启动: {{.AgentName}}"
    - type: exec
      command: "echo 'started' >> /tmp/agent-audit.log"
  on_complete:
    - type: log
      message: "代理完成: {{.AgentName}}"
  on_error:
    - type: exec
      command: "echo 'error' >> /tmp/agent-audit.log"

# --- 独立 MCP 服务器 ---
mcp_servers:
  - name: workspace
    command: mcp-filesystem
    args: ["/home/user/project"]
  - name: database
    command: mcp-sqlite
    args: ["--db", "/data/app.db"]
```

### 3.2 三种执行模式

#### Spawn 模式（默认）

创建**独立的 Runtime 实例**，拥有自己的会话、工具集和配置。与父级代理完全隔离。

```
编排者 Runtime
    ↓ 调用 agent_code_writer
AgentTool.executeSpawn()
    ├─ 创建受限工具集（排除 agent_*）
    ├─ 创建独立 Runtime + 配置覆盖
    ├─ captureChannel 捕获输出
    └─ 返回最终输出
```

#### Fork 模式

继承父级 Runtime 的**完整会话上下文**（消息历史 + 系统提示词），只追加新任务指令。适用于需要理解当前对话上下文的子任务。

```
编排者 Runtime（含消息历史）
    ↓ 调用 agent_analyzer（fork 模式）
AgentTool.executeFork()
    ├─ RuntimeFromContext 获取父级 Runtime
    ├─ CheckForkDepth 检查递归深度（最大 3 层）
    ├─ 创建 SubagentContext（隔离 + 继承）
    ├─ 子 Runtime 继承父级上下文
    └─ 追加 <fork-directive> 指令
```

**递归深度防护**：Fork 嵌套最多 3 层（`MaxForkDepth = 3`），超过则返回错误。

#### Background 模式

**异步执行**，立即返回代理 ID。调用者可以稍后通过 `BackgroundManager` 查询结果。

```
编排者
    ↓ 调用 agent_researcher（background 模式）
AgentTool.executeBackground()
    ├─ BackgroundManager.Spawn() 启动 goroutine
    ├─ 立即返回 agent ID
    └─ 编排者继续执行其他任务

    ... 后台 goroutine 执行 ...

编排者
    ↓ BackgroundManager.GetResult(agentID) 查询结果
    └─ 获取最终输出
```

### 3.3 SubagentContext — 隔离上下文

`SubagentContext` 精确控制子代理的共享和隔离边界。

**文件位置**：`internal/agent/subagent_context.go`

```
SubagentContext
├─ 隔离层
│   ├─ ToolRegistry    受限工具集（agent_* 始终排除）
│   ├─ Permission      权限模式
│   ├─ Cancel          独立取消控制
│   └─ AbortOnParent   是否跟随父级取消
├─ 继承层（只读引用）
│   ├─ ParentMessages  父级消息历史（仅 fork 模式）
│   ├─ SystemPrompt    父级系统提示词
│   ├─ Memory          共享记忆存储
│   ├─ Sessions        共享会话管理器
│   └─ DB              共享数据库
└─ 追踪层
    ├─ AgentID         唯一标识
    ├─ ParentID        父级代理 ID
    ├─ Depth           嵌套深度（0 = 顶层）
    └─ ChainID         调用链 ID
```

**上下文传递机制**：通过 `context.Context` 在代理调用链中传递：

```go
// 存储/获取父级 Runtime
ctx = RuntimeToContext(ctx, parentRuntime)
parent := RuntimeFromContext(ctx)

// 存储/获取子代理上下文
ctx = SubagentContextToCtx(ctx, subCtx)
sc := SubagentContextFromCtx(ctx)
```

### 3.4 AgentOrchestrator — 并行/DAG 调度

**文件位置**：`internal/agent/orchestrator.go`

支持两种调度模式：

#### 并行执行

所有任务同时执行（受 `maxParallel` 限制，默认 4）。单个任务失败不中断其他任务。

```go
orch := NewAgentOrchestrator(agentMgr, 4)
results, _ := orch.ExecuteParallel(ctx, tasks, executor)
```

#### DAG 依赖调度

按依赖关系拓扑排序，逐层并行执行：

```go
tasks := []AgentTask{
    {ID: "a", AgentName: "researcher", Task: "搜集资料"},
    {ID: "b", AgentName: "analyzer", Task: "分析数据", DependsOn: []string{"a"}},
    {ID: "c", AgentName: "writer", Task: "撰写报告", DependsOn: []string{"a"}},
    {ID: "d", AgentName: "reviewer", Task: "审核报告", DependsOn: []string{"b", "c"}},
}
// 执行顺序：a → [b, c]（并行）→ d
results, _ := orch.ExecuteDAG(ctx, tasks, executor)
```

**拓扑排序**：Kahn 算法，自动检测循环依赖并返回错误。

### 3.5 BackgroundManager — 后台异步管理

**文件位置**：`internal/agent/background.go`

| 方法 | 说明 |
|------|------|
| `Spawn(ctx, spec, runner)` | 启动后台 goroutine，立即返回 agent ID |
| `GetResult(agentID)` | 非阻塞查询结果（running 返回 nil） |
| `Wait(ctx, agentID)` | 阻塞等待完成 |
| `Cancel(agentID)` | 取消正在运行的后台代理 |
| `List()` | 列出所有后台代理状态 |
| `Cleanup()` | 清理已完成的代理记录 |
| `NotifyCh()` | 聚合通知 channel（running→completed/failed/cancelled） |

**状态机**：

```
Spawn() → Running → Completed（成功）
                   → Failed（错误）
                   → Cancelled（被取消）
```

### 3.6 PromptCache — 系统提示词缓存

**文件位置**：`internal/agent/prompt_cache.go`

当多个子代理共享相同配置时，系统提示词只构建一次并缓存复用。减少 LLM API 调用中的提示词重复构建开销。

```go
cache := NewPromptCache()
prompt := cache.GetOrBuild(cacheKey, func() string {
    return buildExpensivePrompt() // 仅首次调用
})
```

**缓存键**：由 `AgentSpec.Name + Model + sha256(SystemPrompt)` 组合生成，确保相同配置的代理共享缓存。

**统计接口**：

```go
stats := cache.Stats()
// stats.Size      = 缓存条目数
// stats.TotalHits = 总命中次数
```

### 3.7 SidechainRecorder — 执行历史记录

**文件位置**：`internal/agent/sidechain.go`

每个子代理的执行历史独立于主对话记录，支持：
- **审计**：谁在什么时间执行了什么操作
- **恢复**：从 Sidechain 重建子代理状态
- **调试**：追踪整个调用链的执行过程

**条目类型**：

| 类型 | 说明 |
|------|------|
| `message` | 消息（user/assistant） |
| `tool_call` | 工具调用 |
| `tool_result` | 工具执行结果 |
| `status` | 状态变更 |

**双存储实现**：

| 存储 | 类 | 适用场景 |
|------|---|---------|
| 文件存储 | `FileSidechainStore` | 调试、轻量使用（`~/.ironclaw/sidechains/`） |
| SQLite 存储 | `SQLiteSidechainStore` | 生产环境（表：`sidechain_entries`） |

**恢复能力**：

```go
entries, _ := RecoverFromSidechain(store, agentID)
// 重建子代理的完整执行历史
```

## 四、安全与隔离

### 4.1 权限分级模型

**文件位置**：`internal/agent/permission.go`

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| `bypass` | 允许所有操作 | 受信任的只读代理（如规划器） |
| `accept_edits` | 自动批准读写，拦截危险操作 | 代码编写代理 |
| `bubble` | 所有请求发送到父级审批 | 敏感操作代理 |
| `""` (默认) | 遵循标准权限检查 | 普通代理 |

**危险操作黑名单**（`accept_edits` 模式下自动拦截）：

```
rm -rf, rm -r /, chmod 777, kill -9, killall, pkill,
dd if=, mkfs, fdisk, iptables, ufw, firewall-cmd,
shutdown, reboot, halt, :(){ :|:& };:
```

**Permission Bubble 机制**：

```
子代理需要执行危险操作
    ↓
PermissionEvaluator.Check()
    ↓ 发送到 parentCh
父级 Runtime 收到 PermissionRequest
    ↓ 用户审批
PermissionResponse{Allowed: true/false}
    ↓ 通过 ResponseCh 返回
子代理继续/拒绝执行
```

### 4.2 工具隔离

所有子代理的工具集都经过 `buildScopedRegistry()` 过滤：

1. **agent_* 工具始终排除** — 防止子代理递归调用其他子代理
2. **白名单优先** — 如果 `spec.Tools` 非空，仅包含列出的工具
3. **独立注册表** — 每个子代理拥有独立的 `tool.Registry` 实例

### 4.3 递归防护

- Fork 模式最大嵌套深度：**3 层**（`MaxForkDepth = 3`）
- agent_* 工具排除确保 Spawn 模式子代理无法调用其他子代理
- `CheckForkDepth()` 在每次 fork 前检查深度

## 五、执行后端

**文件位置**：`internal/agent/backend.go`

| 后端 | 类型 | 说明 | 状态 |
|------|------|------|------|
| `InProcessBackend` | goroutine | 零开销，默认后端 | ✅ 已实现 |
| `SubprocessBackend` | os/exec | 进程级隔离，需要二进制文件 | 🔲 占位 |
| `DockerBackend` | 容器 | 完全隔离（文件系统、网络） | 🔲 占位 |

**自动降级**：如果请求的后端不可用，自动降级到 `InProcessBackend`。

```go
backend := SelectBackend(spec.Backend, executor)
// DockerBackend 不可用 → 自动降级到 InProcessBackend
```

## 六、生命周期钩子

**文件位置**：`internal/agent/agent_hooks.go`

子代理在执行的关键节点触发钩子：

```
OnStart      → 代理启动时
OnToolCall   → 每次工具调用前
OnComplete   → 代理成功完成时
OnError      → 代理出错时
OnTimeout    → 代理超时时
```

**钩子类型**：

| 类型 | 行为 |
|------|------|
| `log` | 通过 slog 输出结构化日志 |
| `exec` | 执行 shell 命令（10 秒超时） |

**重要特性**：钩子失败**不会阻塞**代理执行，仅记录警告日志。

## 七、Per-Agent MCP 服务器

**文件位置**：`internal/agent/agent_mcp.go`

每个子代理可以声明自己的 MCP 服务器。代理启动时初始化，结束时清理。

```yaml
# 在代理规格中配置
mcp_servers:
  - name: workspace
    command: mcp-filesystem
    args: ["/home/user/project"]
    requires_approval: false
  - name: database
    command: mcp-sqlite
    args: ["--db", "/data/app.db"]
    requires_approval: true
```

**生命周期**：

```
AgentTool.Execute()
    ├─ AgentMCPManager.InitializeForAgent() → 启动 MCP 服务器 + 注册工具
    ├─ ... 代理执行 ...
    └─ AgentMCPManager.CleanupForAgent()    → 停止服务器 + 清理工具
```

## 九、Gateway 初始化顺序

在 `internal/gateway/gateway.go` 中，子代理系统的初始化顺序：

```
1. 创建 AgentManager
2. 加载代理规格（~/.IronClaw/agents/*.yaml + 额外目录 + 内联定义）
3. 创建 BackgroundManager → 注入 AgentManager 和 Runtime
4. 创建 PromptCache → 注入 Runtime
5. 创建 AgentMCPManager → 注入 AgentManager 和 Runtime
6. 创建 AgentOrchestrator → 注入 Runtime 和 CognitiveAgent
7. AgentManager.RegisterAll() → 创建 AgentTool 实例并注册到工具注册表
8. Runtime.SetAgentManager() → 使编排者可以发现子代理
```

## 十一、常见使用场景

### 场景 1：代码编写 + 审核流水线

```yaml
# researcher.yaml
name: researcher
description: 搜索相关代码和文档
execution_mode: fork        # 继承对话上下文
permission_mode: bypass     # 只读操作
tools: [file_read, bash]
```
