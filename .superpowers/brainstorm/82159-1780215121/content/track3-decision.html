轨道 3 · 架构决策 · core/ 包去留

这是整个修复计划里影响最大的单一决策。需要你确认。

A

归档 core/ → internal/archived/core/

保留代码作为参考实现，但从生产路径完全移除。boot/ + adapter/ 一并归档。

Pros

  • 消除两套运行时的认知负担
  • 删 ~2500 行未使用代码
  • 新人不再困惑"该用哪个"

Cons

  • 丢掉了一个设计更好的实现
  • 未来如需迁移要从零开始
B

启动迁移：agent/ → core/

以 core/ 为目标架构，逐步将 agent.Runtime 和 CognitiveAgent 迁移到 core/ 的 middleware 模型。

Pros

  • core/ 设计确实更好（middleware、事件总线、小接口）
  • 一次投入，长期受益
  • Gateway 自然瘦身

Cons

  • 风险最大——agent/ 有 80+ 源文件
  • 迁移期间两套并行维护
  • 可能需要 2-3 个额外 worktree 迭代
C

混合方案：Gateway 用 core/，Agent 保留 agent/

Gateway 层的简单任务（无状态 LLM 调用、tool dispatch）切到 core.Agent；CognitiveAgent 的 5 阶段循环保留在 agent/。

Pros

  • 渐进式迁移，风险可控
  • Gateway 立即受益于 core/ 的简洁性
  • cognitive 复杂逻辑不受影响

Cons

  • 仍然两套运行时，只是分工更清晰
  • 中间态可能持续很久（"temporary = permanent"）
我的推荐: C（混合方案）

理由很实际——agent/ 的 CognitiveAgent（1562行）太复杂，迁移风险不值得现在承担。
但 Gateway 里很多"简单 LLM 调用 + tool dispatch"的场景（如 SubAgent 执行、eval harness）
用 core.Agent 的 middleware 模型完全够用，而且代码量减半。

具体做法: 修改 SubAgentManager.Spawn() 和 TaskCoordinator 的底层执行器，
从 agent.Runtime → core.Agent，保留 CognitiveAgent 不变。
结果是 core/ 不再"零使用"，Gateway 顺便瘦身，agent/ 的复杂部分保持不变。
