IronClaw 全线修复 — 三种实施策略

P0 即时修复 → P1 测试+拆分 → P2 重构 → 架构债务。跨 20+ 改动点，涉及 ~40 个文件。

A

顺序推进 · 按优先级串行

P0 → P1 → P2 → 架构债务，一个 worktree 顺序执行。安全、可预测、但慢。

时间估算

P0: 30min → P1: 3-4h → P2: 4-6h → 架构: 6-8h ≈ 总计 14-19 小时

Pros

  • 每一步验证后再前进
  • 不会产生合并冲突
  • 审阅友好（每个阶段一个 PR）

Cons

  • 周期最长
  • 大量串行等待
  • 后期阶段可能因前期设计决定而返工
B

并行 Worktree · 按模块独立推进

按受影响模块拆成独立 worktree，互不依赖的并行执行。快但需要精心协调合并顺序。

Worktree 拆分（3 条并行轨道）

轨道 1 "外科手术" — P0 三修 + gateway 拆分 + cognitive 拆分
轨道 2 "测试补全" — 9 个零测试包全部补测试
轨道 3 "架构决策" — core/ vs agent/ 去留 + config 简化 + TUI 拆分

Pros

  • 总耗时最短（3 轨并行 ≈ 6-8h）
  • 互不阻塞
  • 每条轨道可以独立 review

Cons

  • 合并时需要处理冲突
  • 轨道 1 和 2 可能 touching 相同文件
  • 需要明确的合并顺序规划
C

垂直切片 · 按子系统逐个击破

每个子系统从 P0→P2→架构一次性做完。E.g. 先彻底搞定 agent/，再搞定 gateway/，再搞定 knowledge/。

切片顺序

Slice 1 "Agent 核心" — cognitive.go 拆分 + act.go 拆分 + agent 测试 + core/ 归档
Slice 2 "Gateway 编排" — context 泄漏 + god object 拆分 + config 简化
Slice 3 "子系统补强" — knowledge/wasm/code_engine/guardian 测试 + TUI 拆分

Pros

  • 每个切片独立可交付
  • 风险隔离好
  • 自然的分批 PR 节奏

Cons

  • 模块间依赖可能导致返工
  • 中等耗时（≈ 10-14h）
  • 需要先锁定模块间接口变更
